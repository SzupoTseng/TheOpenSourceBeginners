## 序　當「開源」多到會淹死人

二〇二六年，打開 GitHub 的 Trending 頁面，像是站在消防栓前喝水。

每天有上千個新專案掛著「顛覆」「殺手」「下一代」的標題衝上熱榜；每一個都號稱解決了你的痛點，每一個的 README 都閃閃發光。可是當你真的要在週一早上的架構會議上拍板「我們用哪個」時，那種資訊過載會瞬間變成一種近乎生理性的焦慮——**紅的不一定適合你，適合你的你根本沒聽過，而你聽過的可能三個月後就停更了。**

這本書，就是寫給站在這座資訊消防栓前的你。

它不追熱點、不打廣告，只做一件事：把當前開源界**最紅、最多人用、最新熱度**的 196 個專案，一個一個攤開，用同一副手術刀解剖，讓你看清每一個專案**紅的真正原因、它的技術核心到底硬在哪、以及它哪裡會在半夜咬你一口**。

> 💡 全書的核心命題
> **選型的盡頭，不是「用最紅的」，而是「看懂它為什麼紅、以及它哪裡會咬你」。**

### 三個面向：你在為哪一種「紅」買單

市面上所有「熱門開源排行」都混為一談，但**紅**其實有三種完全不同的臉：

| 面向 | 符號 | 它回答的問題 | 典型代表 |
|------|------|------------|---------|
| **最紅面向** | 🏆 | 聲量最大、star 最猛、社群最狂熱 | React、freeCodeCamp、Next.js |
| **最多人用** | 👥 | 沉默但無所不在的產業地基 | Linux Kernel、PostgreSQL、Nginx |
| **最新熱度** | 🔥 | 正在改寫遊戲規則的新星 | Ollama、DeepSeek-R1、LangGraph |

搞混這三者，是新手選型最大的陷阱：拿「最新熱度」的實驗性框架去扛「最多人用」才該承擔的生產流量，等於拿賽車引擎去跑貨運。本書每個專案都標了面向，就是要你先問自己——**我現在需要的，是聲量、是穩定，還是未來？**

### 每一景，同一副骨架

為了讓 196 個專案可以被公平地橫向比較，每一個都用同一套結構拆給你看：

> **標籤 → GitHub 體檢 → 起源 → 技術核心 → 解決的痛點 → 理論基礎 → 在 AI Agent 時代的角色 → 新人須知（大廠第一週）→ 優點／罩門 → 競品對照 → 效益**，最後收在一句「君之一席話」與一段「🔍 老手視角」。

其中兩個欄位是本書對讀者的偏心：

- **技術核心**不會停在行銷話術。我們會直接點名底層機制——LSM-tree 還是 B+tree、PagedAttention 還是 continuous batching、編譯期消滅 Virtual DOM 還是執行期 diff。**素材是骨，硬核細節是肉。**
- **新人須知（大廠第一週）**是這本書的私心。它站在「一個剛進大公司 IT 部門的新人」的角度，告訴你：你到職第一週會在哪個環節撞見這個東西、最少要會哪兩三招、以及**學長不會明講、但你一定會踩的那個雷**。

### 關於 star 數與維護者數

每個專案都附了「GitHub 體檢」：star 數、核心維護者、貢獻者、授權、主語言。**請把這些數字讀成「量級」，不是「即時報價」**——開源世界每天都在變，本書的數字是撰寫時的概估，用「約」標示，只為讓你分辨「這是十萬 star 的地基，還是三千 star 的新芽」。需要精確的當日數字，GitHub API 一行就能刷新。

### 一點立場說明

書中所有「老手視角」的點評，都出自一個**中性的資深工程師**視角——不綁定任何公司、任何職級。我們在乎的從來不是「某大廠怎麼說」，而是「一個踩過夠多坑的人，會怎麼替你把關」。

技術會過時，框架會改名，這 196 個專案裡有一半可能三年後就換了主角。但這本書真正想留給你的，不是一份會過期的排行榜，而是一種**看穿熱度、直達本質的選型眼光**。

翻過這一頁，我們先從最基本、卻最多人跳過的一課講起——**怎麼閱讀「紅」這件事。**
