# 《開源器識》寫作規格 — 作者契約（STYLE_SPEC）

> 這份檔案是本書所有章節（含由子代理撰寫的部分）必須嚴格遵守的統一規格。
> 目標：以《代理覺醒 The Agent Awakens》的風格與骨架，把 os1.html 蒐集到的 196 個開源專案，寫成一本**繁體中文**、**技術深度扎實**的選型聖經。

---

## 0. 三條鐵律（違反即重寫）

1. **繁體中文**。技術名詞、專案名、指令、程式碼保留原文（如 `unshare()`、Rollup、eBPF）。
2. **絕對不出現 Google L7 / Staff / FAANG 主管視角這類身分**。素材原文大量使用「在 Google 我們…」「Google L7 Staff 評點」——**一律移除、改寫**成中性的「資深架構師 / 老手」視角。不指名任何具體僱主或職級。第一人稱可用「我」，但不得綁定到某公司。
3. **補強技術深度**。不要只是翻譯素材。每個真實專案都要加入素材沒寫、但業界公認的硬核細節：核心資料結構、關鍵演算法、協定、記憶體/併發模型、時間複雜度、實測量級、典型故障模式。素材是骨，你的技術知識是肉。

## 1. 書名與定位

- 書名：**《開源器識 · The Open Source Beginners》**
- 副標：一本從新手到架構師的開源選型聖經——2026 年最紅、最多人用、最新熱度的 196 個開源專案深度解剖。
- 核心命題：**「選型的盡頭，不是『用最紅的』，而是『看懂它為什麼紅、以及它哪裡會咬你』。」**
- 讀者：要做技術選型的工程師、想補底層原理的轉職者、評估開源供應鏈的架構師。

## 2. 全書骨架（檔案結構，純 .md）

```
00_序言.md
01_如何閱讀本書_三個面向.md        （首篇：最紅／最多人用／最新熱度 的判讀學）
05..23_<slug>_<中文標題>.md         （13 個「篇」切成 19 個章節檔，每檔含數個專案）
附錄A_專案總索引.md                 （196 專案依編號＋一句話定位＋標籤）
附錄B_三面向速查.md                 （🏆最紅／👥最多人用／🔥最新熱度 各自清單）
附錄C_依特性分組索引.md             （程式語言／前端／AI／Infra／資料庫… 分類導航）★使用者指定
附錄D_資深視角的選型現實校準.md      （對「最紅≠最適合」的冷水與鐵律）
```

篇（Part）＝13 個特性分類；每篇開頭一行 `# 第N篇　<分類名>`。

---

## 3. 每個「專案」的統一骨架（這是本書的原子單位）

每個專案是一個 `##` 小節。標題用**全書連續三位數編號**（如 `001`、`047`），這個編號同時是附錄 A 總索引的鍵。每章開頭有 `# 第N篇　<分類>` 的篇名。順序與標題**逐字固定**如下：

```markdown
## 001　ProjectName — 一句話霸氣定位

**標籤**：`#分類` `#語言` `#關鍵技術` `#定位` `#生態` …（至少 5 個，中英不拘，用反引號包）
**面向**：🏆 最紅｜👥 最多人用｜🔥 最新熱度（擇一或並列；依素材標記）
**GitHub 體檢**：⭐ 約 XXk｜核心維護者 約 N 人｜貢獻者 X,000+｜授權 MIT｜主語言 Rust
（★量級估計，非即時數據，以「約」表示；未知或新專案標「資料不詳」。禁止捏造精確數字。）

**起源**：誰、哪一年、為了解決什麼而生。要有故事感，一段。

**技術核心**：★補強重點★。用 2–4 句素材＋你補的硬核細節。要點名真正的機制
（資料結構／演算法／協定／編譯或執行模型），不要停在行銷話術。

**解決的痛點**：它到底消滅了工程師哪一種具體的痛。

**理論基礎**：相關論文、範式或方法論（有才寫，如 CAP、Raft、ReAct、費曼學習法）。

**在 AI Agent 時代的角色**：這個專案怎麼和 LLM／Agent 結合（可沿用素材的想像，但要具體、可信）。

**新人須知（大廠第一週）**：★大廠 IT 新人必讀角度★ 三件事——
①你到職後會在哪個環節撞見它；②最少要會的 2–3 個操作或概念；③新人最常踩的雷。
語氣像學長帶新人，具體、口語、能立刻上手。

**優點 / 罩門**：兩三句優點，兩三句真實缺陷（效能天花板、運維成本、授權陷阱、邊緣 bug）。

**競品對照**：一張 Markdown 表，2–4 個對手，欄位固定為 `| 對手 | 定位 | 相對優勢 | 相對劣勢 |`。

**效益**：選它／學它對企業與個人帶來什麼可量化或可感的價值。

> 💡 君之一席話
> 一句（至多兩句）詩性、犀利、可被引用的 take-away。這是全書的靈魂句。

> 🔍 老手視角──真正的門道
> 用中性資深視角（**不得出現 Google/L7**）講素材裡「主管評點＋商業切入點」的精華：
> 這東西紅的真正原因、大廠選型時真正在看什麼、以及一條可落地的商業機會或反直覺提醒。
```

長度：每個專案約 **650–1000 繁體中文字**（不含表格）。真實大專案（K8s、PyTorch…）可到上限；小工具（Zustand、Hono…）取下限。

**GitHub 體檢的誠實原則**：star／維護者數以「約」給量級（例：freeCodeCamp 約 400k★、K8s 約 110k★、VS Code 約 165k★）。冷門或疑似 AI 生成的專案（如 MindOS-*、SGLang-Kernel）標「資料不詳／新興專案」，不得編造。全書開頭與附錄需聲明數據為量級參考、可用 GitHub API 刷新。

## 4. 語氣與句法

- 對照《代理覺醒》：**畫面感、具體數字、戲劇張力、詩性收尾**。範例句風：「它聰明地把最耗時的模組解析工作外包給瀏覽器內核，自己只當流量的調度員。」
- 用**粗體**打亮關鍵詞（骨架欄位名、關鍵術語）。
- 表格用 sans 風格短句，不要長段。
- `💡` 框永遠是「君之一席話」（金框概念）；`🔍` 框永遠是「老手視角」（藍綠點評框）。與《代理覺醒》的 `_build.py` 樣式相容（雖然本書只交付 .md，仍保留這兩個 emoji 慣例）。

## 5. 補強技術深度：可用的「加料」清單（依專案取用）

- **資料庫**：儲存引擎（LSM-tree vs B+tree）、MVCC、WAL、隔離級別、複製協定、分片、查詢優化器。
- **雲原生**：控制迴圈、宣告式 API、CRD、CNI/CSI、eBPF、Sidecar、共識（Raft/etcd）。
- **AI 推理**：KV-Cache、PagedAttention、continuous batching、量化（GGUF/AWQ/GPTQ）、張量／流水線並行、FlashAttention。
- **前端**：ESM、HMR、Tree-shaking、Virtual DOM vs 編譯期、Islands、RSC、原子化 CSS。
- **語言/執行環境**：JIT、GC、事件迴圈、零成本抽象、所有權模型（Rust）、AOT。
- **訊息/串流**：至少一次/精確一次語意、消費者組、分區、背壓、水位線（watermark）。
- 每處「加料」都要**準確**；不確定的數字用量級描述（「毫秒級」「數萬台」），不要編造精確 benchmark。

## 6. 常見禁區

- ❌ 出現「Google L7」「Staff Engineer」「在 Google 我們」「大廠主管」等身分綁定。
- ❌ 整段照抄素材（素材是繁簡混雜、句子破碎的搜尋殘骸，必須重寫為通順繁中）。
- ❌ 專案標籤少於 5 個。
- ❌ 缺少 `💡 君之一席話` 或 `🔍 老手視角` 任一框。
- ❌ competitor 表欄位改名或缺漏。
